1. System Overview
The SWIFT Compliance Platform is a multi-tenant SaaS application that helps banks manage their SWIFT Customer Security Controls Framework (CSCF) v2025 compliance assessments. The platform enables evidence collection across 8 domains, maps 53 canonical evidence items to 32 CSCF controls, and orchestrates a 5-phase compliance workflow from setup through final KYC-SA submission.

3. Application Workflow (7-Phase Model)
The original UI plan defined a simple status model (not_started, in_progress, under_review, attested, archived). The database schema implements a richer 7-phase lifecycle via the assessment_phase enum: setup ΓåÆ collection ΓåÆ review ΓåÆ approval ΓåÆ reporting ΓåÆ submitted ΓåÆ archived. Phase transitions are captured in snapshot_data for auditability.
3.1 Phase 1: Setup
Platform Admin creates a Tenant with bank details and BIC code. Bank Admin (compliance_officer) is invited and creates an Assessment Cycle, selecting the Architecture Type (A1ΓÇôB). The system auto-generates control_applicability records (32 rows) based on architecture_applicability arrays in the controls table. The cycle enters the 'collection' phase.
3.2 Phase 2: Evidence Collection
The Compliance Officer sees a domain-centric dashboard showing 8 domain cards with completion percentages and a control sufficiency heatmap. IT SMEs upload evidence per item: file attachments (PDF, PNG, XLSX, config exports) stored in GCS, plus form data stored as JSONB in evidence_submissions.form_data. The system supports 4 collection models: standard (one per item), per_system (one per SWIFT host), per_vendor (one per vendor from vendor_registry), and per_quarter (e.g., C5 access reviews). Each submission is versioned; resubmission after return increments the version. AI processing (extraction, summary, classification, control suggestions) runs on file upload via the processing pipeline.
3.3 Phase 3: Review
Evidence submissions enter a 3-level review workflow. L1 (Completeness): Checks that all required fields are filled and files are in correct format. L2 (Quality): Assesses whether the evidence demonstrates control effectiveness. L3 (Assessment): Independent validation by the external assessor for regulatory alignment. Each level creates a review_assignments record with SLA tracking (sla_hours, is_overdue). Reviewers can approve, return (with notes), or escalate. Review comments support threaded discussions with @mentions, attachments, and resolution tracking.
3.4 Phase 4: Approval
4 sequential approval gates must be cleared: (1) evidence_complete ΓÇö all evidence items submitted; (2) internal_review ΓÇö all L2 reviews passed; (3) assessment_complete ΓÇö L3 independent assessment done; (4) final_attestation ΓÇö CISO/Head of Compliance sign-off with MFA confirmation. Each gate tracks prerequisites_met, blocking_items (JSONB), and approver identity. The final gate requires mfa_confirmed = true.
3.5 Phase 5: Reporting
The system generates a compliance report with sections stored as JSONB in assessment_reports. Reports have draft and final types, are versioned, and include a full snapshot_data capture of the assessment state at generation time. Report content includes executive summary, scope statement, methodology, per-control assessments, gap analysis, evidence index, and glossary. AI generates draft content; the Compliance Officer reviews and edits. Final reports are immutable and can be exported to PDF, DOCX, or XML.
3.6 Phase 6ΓÇô7: Submitted & Archived
After the report is finalized, the cycle moves to 'submitted' (KYC-SA submission completed) and eventually 'archived' for historical reference. Year-over-year comparison is enabled via the previous_cycle_id self-reference on assessment_cycles.

4.3 Detailed Table Definitions
4.3.1 tenants
Root entity for multi-tenant isolation. Every other table references tenant_id. RLS policies enforce data separation at the database level.
4.3.2 users
All platform users. Auth handled externally (Firebase Auth / Auth0). This table stores profile, role, and tenant association. Email is unique per tenant, not globally.
4.3.3 assessment_cycles
One record per annual compliance cycle per tenant. Central coordination entity linking evidence, reviews, approvals, and reports. Supports year-over-year comparison via previous_cycle_id self-reference.
4.3.4 evidence_submissions
Core mutable table. One record per evidence item per cycle (or per scope_key variant). The scope_key pattern supports 4 collection models: standard (__default__), per_system (hostname), per_vendor (vendor UUID), per_quarter (Q1_2025). Form data is stored as JSONB, eliminating the need for a separate evidence_form_data table.
4.3.5 evidence_attachments
File metadata for uploaded evidence documents. Actual files stored in GCS versioned bucket. Files are never overwritten (new uploads = new records). Soft-delete only via is_deleted flag.
4.3.6 Reference Tables (Immutable)
audit_frameworks: Framework definitions. Phase 1 has one row for SWIFT CSCF v2025 (32 controls, 25 mandatory, 7 advisory). Includes metadata JSONB with objectives, principles, attestation window.
controls: 32 CSCF controls with framework FK, control_id_code (e.g. '1.1'), control_type (mandatory/advisory), objective, principle, description, implementation_guidance, and architecture_applicability (TEXT[] array like '{A1,A2,A3,A4,B}').
evidence_domains: 8 evidence collection domains (AΓÇôH) with framework FK, display ordering, UI color codes, and total item counts.
canonical_evidence_items: 53 evidence items with JSONB input_schema (defines the UI form per item), JSONB sufficiency_dimensions (per-dimension scoring criteria), collection_model enum (standard/per_system/per_vendor/per_quarter), reuse_tier enum (foundational through control_specific), and collection_priority enum (critical/high/medium).
item_control_mappings: Junction table mapping evidence items to controls. Includes is_primary flag, weight (NUMERIC 0ΓÇô1), and sufficiency_requirement text describing what the evidence must demonstrate for each mapped control.
cross_domain_dependencies: Dependencies between evidence items across domains (e.g. E1 anti-malware depends on A2 system inventory). Includes validation_rule and impact_description.
4.3.7 Workflow Tables
control_applicability: Per-cycle scoping of which controls apply based on architecture type. Auto-generated at cycle creation. Can be manually overridden (override_by user FK, na_justification text).
vendor_registry: Domain F vendor registry per cycle. Each vendor becomes a scope_key for F1ΓÇôF4 evidence submissions. Tracks vendor_name, swift_classification enum, service_description, components_managed (TEXT[]), access_type enum, and contract dates.
sufficiency_scores: Per-control aggregate score per cycle. Recomputed when evidence changes. Drives real-time compliance sidebar. Includes gap_details and ai_gap_analysis JSONB.
sufficiency_evaluations: Append-only per-item evaluation results. Each run stores input_completeness_pct, content_validation_result, dimension_scores (per-dimension pass/fail/partial), overall_score, gaps_identified, and recommendations.
review_assignments: 3-level review workflow (L1 completeness, L2 quality, L3 assessment). Tracks assignment, status, decision, SLA hours, overdue flag. Completed reviews are immutable.
review_comments: Threaded discussion on evidence submissions. Supports @mentions (UUID[]), thread resolution (is_resolved, resolved_by, resolved_at), comment attachments, and parent_comment_id self-reference for threading.
approval_gates: 4 sequential gates per cycle: evidence_complete, internal_review, assessment_complete, final_attestation. Tracks prerequisites_met, blocking_items (JSONB), mfa_confirmed (required for final gate).
assessment_reports: Generated reports with draft/final types, versioning, JSONB sections array, compliance_status, GCS export path, and full snapshot_data capture.
4.3.8 Audit Log (Immutable, Partitioned)
The single most important table. Every state change across every entity is recorded. INSERT-ONLY enforced by database trigger. Partitioned by month (RANGE on timestamp) with 12 monthly partitions. Columns: tenant_id, timestamp, actor_id, actor_role, actor_ip (INET), action, action_category (ENUM with 14 categories), entity_type, entity_id, before_state/after_state/diff (JSONB), metadata, cycle_id (denormalized), session_id.

6. Authentication & Authorization
6.1 Auth Flow (Revised)
User authenticates with Firebase Auth / Auth0 (external provider) and receives an external token. The platform API exchanges this for an internal JWT. The middleware chain: (1) Validate JWT signature + expiry, (2) Extract userId + tenantId + role from claims, (3) SET LOCAL app.tenant_id = '{uuid}' for RLS enforcement, (4) Role guard checks endpoint permissions, (5) Audit middleware logs the action to audit_log.
6.2 JWT Claims
Claims: sub (user UUID), email, role (user_role enum value), tenantId (UUID), mfaVerified (boolean), iat, exp. Token lifetime: 24 hours for access token, 30 days for refresh token.
6.3 Multi-Tenancy via RLS
All 13 tenant-scoped tables have RLS enabled with policies checking tenant_id = current_setting('app.tenant_id', true)::uuid. The API middleware must call SET LOCAL app.tenant_id = '{uuid}' before every query within a transaction. The application database user must NOT be a superuser (superusers bypass RLS). Platform admins bypass RLS by temporarily setting app.tenant_id to the target tenant.

8.1 AI Processing Pipeline
File Upload ΓåÆ upload_status: 'uploading' ΓåÆ 'uploaded' ΓåÆ 'processing' (AI extraction in progress) ΓåÆ 'processed' (AI analysis complete) or 'failed' (processing error). The pipeline: (1) Text extraction / OCR, (2) Chunking + vectorization, (3) Control mapping via similarity search against control descriptions, (4) Content validation against sufficiency dimension criteria, (5) Results stored on both evidence_attachments (file-level) and evidence_submissions (submission-level).
