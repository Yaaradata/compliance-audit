FILE 3 — AI Signals Catalogue
Each AI signal is a graph-resident AIInsight node (Pass 3 §5.2). Catalogue extends Pass 1's AI opportunities with control-level detection signals.
Signal taxonomy (six classes)

Anomaly — single-event or short-window deviation from learnt baseline
Drift — sustained divergence in process / behaviour over rolling window
Coverage gap — design-time mismatch between obligations and controls
Effectiveness decay — declining CES trend
Evidence quality — declining completeness / timeliness / linkage
Cluster / RCA — pattern across multiple Issues / CIs / Subjects

Signal lifecycle
Every signal flows through:
Detection      →  AIInsight node created (with confidence, model_version, source data refs)
   │
   ├─→ if confidence ≥ τ_action  →  auto-create Issue + ActionTask
   ├─→ if τ_review ≤ confidence < τ_action  →  human review queue
   └─→ if confidence < τ_review  →  silent (logged only)

Human review   →  AIInsight.human_review_status ∈ {accepted, rejected, overridden}
              →  feedback signal → next model training cycle (training_data_snapshot_id pinned)

Resolution     →  if accepted, AIInsight :RESULTED_IN Issue; lifecycle continues
              →  if rejected, AIInsight :RESULTED_IN null; counts as FP for model
Model meta-governance
All models are graph nodes (Model {id, capability, version, trained_at, training_data_id, approved_by, effective_from/to}). Each is in scope for R-MR-001 (model risk) — drift, validation failures, and override patterns produce AIInsight {type=meta} records. The platform's own AI is not exempt from MRM governance.
