# System Capabilities & Data Model — Pass 2

Building on the Pass 1 personas (CRO/CAO, Risk/Compliance Leadership, Compliance Officers/Audit Managers), this pass defines the platform's core capabilities, the normalized entity model that drives them, and the metrics that link them back to persona success.

A foundational principle from the source model carries throughout: **separate design-time entities from their run-time instances**, and treat every run-time entity as event-sourced so any view can be rewound to a historical date. The RCM is the design-time substrate; the platform's job is to materialise the run-time graph that shadows it.

---

## 1. System Capabilities

Eight core capabilities, each tied to a primary persona and one of the three system goals. A ninth — **Time-Travel & Lineage** — is a foundational platform service that every capability depends on rather than a standalone surface.

| # | Capability | What it does | Primary persona | System goal | RCM grounding |
|---|---|---|---|---|---|
| C1 | **Process Execution Telemetry** | Derives the *actual* process from system event logs (payment hub, core banking, case mgmt, screening engines). Reconciles observed paths against the documented `Process → ProcessStep` model and surfaces variant drift. | Process Owner *(supplies all others)* | Real-time visibility | Wire Payments, Customer Onboarding, AML Alert Disposition, Vendor Onboarding, Model Validation, Loan Origination — the six anchor processes. |
| C2 | **Obligation Decomposition & Mapping** | Parses regulatory text (OCC bulletins, FFIEC manuals, FCA Handbook, RBI Master Directions) into atomic, testable `Obligation` records and maps each to one or more `Control` records with a coverage strength. | Risk / Compliance Leadership | Regulatory traceability | The 51 obligations (`OBL-BSA-*`, `OBL-OFAC-*`, `OBL-SR117-*`, `OBL-OCC-2023-17-*`, etc.) become first-class linked entities, not spreadsheet columns. |
| C3 | **Control Instance Reconciliation** | For every activity in `StepExecution`, asks: *was a control expected to fire here, did it fire, and is its evidence captured?* Surfaces the three failure modes — designed but not operating, operating without evidence, evidence without effectiveness signal. | Risk / Compliance Leadership | 100% auditability | Replaces the static `Last Test Result` field on a control with a continuous signal; e.g., `WP-C001` (dual approval ≥$100K) is reconciled against every wire `StepExecution`. |
| C4 | **Control Effectiveness Monitoring** | Beyond *did it fire?*, asks *did it catch what it was designed to catch?* — using outcome reconciliation, false-positive/false-negative analysis, and override-rate trends. Produces a continuous `effectiveness_score`. | Risk / Compliance Leadership | Real-time visibility | Replaces annual ratings (`Effective with Observations`) with daily signal; e.g., `KRI-FC-003` beneficiary callback completion *worsening* is read as `WP-C003` effectiveness decay. |
| C5 | **Risk Posture Engine** | Aggregates control effectiveness, KRI bands, open-issue load, and appetite breaches into a residual-risk view by domain. Generates the *what changed this week* narrative with attribution. | CRO / CAO | Real-time visibility | Drives the cockpit: 7 risk domains × 15 appetite metrics × 22 open issues × 64 KRIs, rolled up with drill paths in ≤2 clicks. |
| C6 | **Issue Detection & Root-Cause Clustering** | Promotes drift signals (failed `ControlInstance`s, breached KRIs, unusual `ProcessExecution` variants) to candidate `Issue` records. Clusters open issues by causal pattern — capacity, data quality, threshold calibration, vendor delivery — not by category. | Risk / Compliance Leadership | Real-time visibility | The current 22-issue population reveals real clusters: capacity (`ISS-2026-009`, `-031`, `-040`, `-052`), data quality (`ISS-2026-018`, `-046`), threshold calibration (`ISS-2025-079`). Clustering is computed, not authored. |
| C7 | **Population Reperformance & Workpaper Assembly** | Given a control's `design_condition`, replays it against the full in-scope population for any historical window and surfaces exceptions with lineage. Assembles regulator-acceptable workpapers (PCAOB, OCC, FCA, RBI formats) in one export. | Compliance Officer / Audit Manager | 100% auditability | Replaces sampling on `WP-C001`, `CO-C002`, `AML-C002`, etc.; turns the auditor's evidence chase into a query. |
| C8 | **Regulatory Change Impact Analysis** | When a `RegulatoryChangeEvent` is ingested, traces forward through `Obligation → Control → ProcessStep → ProcessExecution` and produces a draft impact assessment with affected entities, coverage gaps, and required actions. | Risk / Compliance Leadership | Regulatory traceability | A new OCC bulletin amending TPRM expectations triggers traversal from `OBL-OCC-2023-17-*` to affected vendor controls (`VO-C001`–`VO-C010`) and exposes coverage gaps. |

**Foundational service — Time-Travel & Lineage.** Every entity is event-sourced. Any view (the CRO's cockpit, the obligation coverage matrix, an audit workpaper) is answerable as of a historical date with full lineage from the displayed value back to the source `StepExecution` and `Evidence` records. This is not a feature; it is a property of the data model.

## 2. Data Model — Entities and Fields

### 2.1 Design principles applied

1. **Design-time vs. run-time separation.** Every "definition" entity (`Control`, `Process`, `Obligation`, `Risk`, `KRI`, `AppetiteMetric`) has a corresponding event-sourced run-time entity (`ControlInstance`, `ProcessExecution`/`StepExecution`, `RegulatoryChangeEvent`, `RiskAssessment`, `KRIObservation`, `AppetiteObservation`).
2. **Every run-time entity is event-sourced.** Append-only log + materialised current state. Time is a queryable dimension.
3. **Stable IDs are normative.** RCM identifiers (`WP-C001`, `OBL-OFAC-001`, `R-OP-001`, `ISS-2026-007`) are preserved verbatim as primary keys.
4. **Multi-value cells become join tables.** No `OBL-001;OBL-002` strings — every M:N relationship is a typed link entity with its own attributes (e.g., coverage strength, position, weight).
5. **Jurisdictional overlay, not jurisdictional fork.** Same entities, with `jurisdiction` as an attribute and obligation lenses applied at the rules layer.

## 3. Relationships

All multi-value cells in the RCM resolve into typed join entities with their own attributes — never into delimited strings.

### 3.1 Core relationships

| From | To | Cardinality | Join entity / Attributes |
|---|---|---|---|
| `Risk` ↔ `Control` | M:N | `RiskControlMitigation` — `coverage_strength` (Primary / Secondary / Contributory), `weight` (0–1) |
| `Control` ↔ `ProcessStep` | M:N | `ControlEmbedding` — `position` (pre / in-line / post), `is_required` |
| `Control` ↔ `Obligation` | M:N | `ObligationCoverage` — `coverage_strength` (Direct / Partial / Contributory), `evidence_expectation_id` |

| `Issue` ↔ `Control` | M:N | `IssueControlLink` — `link_type` (Affected / Root-Cause), `link_strength` |
| `Issue` ↔ `ControlInstance` | M:N | `IssueInstanceLink` — points to the specific failures |
| `Issue` ↔ `Risk` | M:N | `IssueRiskLink` |

```
                                                    │                                     ├──< RiskAssessment

                                                    │                        │            │

                                                    ▼                                                       │
                                              ControlInstance ──────────────────────────────────────────────┘
                                                    │
                                                    ├──< ControlInstanceEvidence >── Evidence
                                                    │
                                                    └──< IssueInstanceLink >── Issue ──< Action
                                                                                  │
                                                                                  └── IssueControlLink, IssueRiskLink

RegulatoryChangeEvent ──ObligationChangeImpact──< Obligation
```

### 3.3 Two design rules these relationships enforce

- **Empirical lineage from any aggregate in ≤2 hops.** A CRO appetite breach drills to `Risk → ControlInstance` (failed instances) → `StepExecution` (the affected activity). Two clicks, not five.
- **Ownership traceability at the instance level.** Every `ControlInstance` carries an `operator_id`; every `Issue` carries an `owner_id`; every `Action` carries an `owner_id`. The "process owner ≠ control owner ≠ control operator" RACI confusion noted in the foundational model becomes structurally explicit, not a forensic exercise.

## 4. Derived Metrics

All three required scores are computed from the entity model above. Each is normalised 0–100 with traffic-light bands, time-windowed, and decomposable so a click on the score reveals its components and ultimately the underlying `ControlInstance` or `Evidence` records.

### 4.1 Control Effectiveness Score (CES)

Computed per `Control` over a rolling window (default 90 days), aggregable up to process or domain.

```
CES = 0.40 × OperatingRate
    + 0.40 × CatchRate
    + 0.20 × EvidenceCompleteness

where over the window:
  OperatingRate         = count(ControlInstance where expected_to_fire AND fired)
                        / count(ControlInstance where expected_to_fire)

  CatchRate             = count(ControlInstance where fired AND caught_what_designed = true)
                        / count(ControlInstance where fired AND caught_what_designed in {true, false})

  EvidenceCompleteness  = count(ControlInstance where fired AND evidence_complete)
                        / count(ControlInstance where fired)
```

Bands: `≥90 = Effective` · `75–89 = Effective with Observations` · `60–74 = Needs Improvement` · `<60 = Ineffective`. A score with insufficient `expected_to_fire` denominators (low population) is flagged `Insufficient Population` rather than scored — preventing false confidence on rare controls.

This replaces the static `Last Test Result` field with a continuously-computed signal of the same type.

### 4.2 Risk Exposure Score (RES)

Computed per `Risk`, aggregable to `risk_domain` and to enterprise. The number the CRO cockpit leads with.

```
RES = InherentBase
    × (1 − ControlMitigationFactor)
    × KRIPressure
    × IssueLoad
    × AppetitePenalty
    × StalenessPenalty

where:
  InherentBase            = mapping of latest RiskAssessment.inherent_rating to base score
                            (High = 80, Medium = 50, Low = 20)
  ControlMitigationFactor = weighted average CES of all Controls linked to this Risk via
                            RiskControlMitigation (weighted by coverage_strength)
                            ∈ [0, 1]
  KRIPressure             = 1 + 0.1 × (count of linked KRIs in amber)
                              + 0.3 × (count of linked KRIs in red)
  IssueLoad               = 1 + 0.05 × (open Medium issues linked)
                              + 0.20 × (open High issues linked)
                              + 0.10 × (issues past due)
  AppetitePenalty         = 1.0 if all linked AppetiteMetrics green
                            1.2 if any amber
                            1.5 if any red
  StalenessPenalty        = 1.0 if latest RiskAssessment.as_of_date < 90 days
                            1.1 if 90–180 days
                            1.3 if > 180 days
```

Bands: `≤30 Green` · `31–60 Amber` · `>60 Red`. Output normalised to 0–100 with a hard cap at 100.

The composition guarantees the CRO's number is *defensible upward* — every multiplier resolves to underlying `ControlInstance`, `KRIObservation`, `Issue`, or `AppetiteObservation` records.

### 4.3 Audit Readiness Score (ARS)

Computed per `Control`, per `Process`, per `Obligation`, or per `Test`. Answers the auditor's question: *"if the regulator asks for evidence on this tomorrow, can I deliver it?"*

```
ARS = 0.30 × EvidenceCoverage
    + 0.25 × PopulationTestability
    + 0.20 × WalkthroughFreshness
    + 0.15 × IssueDrag
    + 0.10 × WorkpaperExportability

where:
  EvidenceCoverage       = % of ControlInstances in scope window with complete,
                           hash-verified Evidence
  PopulationTestability  = % of ProcessExecutions in scope with machine-readable
                           StepExecution telemetry (vs. requiring manual reconstruction)
  WalkthroughFreshness   = function of days_since_last_reconciled_walkthrough
                           (1.0 if ≤30, linear decay to 0.0 at 365)
  IssueDrag              = 1 − (open High-severity Issues linked / threshold)
                           (clamped to [0, 1])
  WorkpaperExportability = 1.0 if all linked Evidence has retention_class
                           and source_system metadata complete; else weighted partial
```

Bands: `≥85 Audit-Ready` · `70–84 Audit-Ready with Gaps` · `<70 Not Audit-Ready`.

This metric is the auditor's KPI for the platform itself: the platform is succeeding when ARS is high and the auditor's evidence chase has been replaced by query.

### 4.4 Two metrics worth adding

Beyond the three required, the model naturally produces two more:

- **Obligation Coverage Score (OCS)** — per `Obligation`, the weighted CES of all Controls linked via `ObligationCoverage`, with a penalty for any obligation linked to fewer than two distinct controls. Drives the Risk Leadership coverage matrix.
- **Process Variant Drift Score (PVDS)** — per `Process`, the divergence between the documented `ProcessStep` chain and the empirical distribution of `StepExecution.variant_signature` over a window. Drives C1 and feeds C6.

## What this enables for the next pass

The combination of capabilities + entities + metrics here is sufficient to generate the next layer of artefacts without further modelling:

- **JSON Schemas / DDL** — directly from §2 entity tables and §3 relationships (each join becomes a table; each enum becomes a check constraint; each event-sourced run-time entity gets an append-only log + materialised view).
- **Persona-by-view UI wireframes** — each Pass 1 success criterion maps to an entity slice: the CRO cockpit reads `RiskAssessment + RES + AppetiteObservation + KRIObservation`; the Compliance Leadership coverage matrix reads `ObligationCoverage + CES`; the auditor workbench reads `ControlInstance + Evidence + TestExecution`.
- **API contracts** — CRUD on design-time entities, append-only event ingestion on run-time entities, query endpoints for the three derived metrics with as-of date parameters.
- **AI capability backlog** — Tier 1 (process mining → C1, control reconciliation → C3, obligation mapping → C2) maps directly to capabilities; Tier 2 (effectiveness analytics → C4, regulatory change impact → C8, anomaly detection → C6) layers on the same data.
