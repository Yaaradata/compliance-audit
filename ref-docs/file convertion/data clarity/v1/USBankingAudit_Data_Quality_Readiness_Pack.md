## 1. Executive Summary

Pass 1 of this initiative answered the foundational question of inventory: what source systems exist, what data they produce, what their join keys are, and what sample records look like across Customer Onboarding, Vendor Onboarding, and Loan Origination. Pass 2 built the correlation engine on top of that inventory, defining how raw source records are assembled into ProcessExecutions, how StepExecutions are sequenced and timed, how ControlInstances are evaluated to pass or fail, and how the Control Effectiveness Score is computed from operating rate, catch rate, and evidence completeness. Pass 3 now addresses the question that determines whether any of those outputs can be trusted: are the underlying data pipelines delivering records that are complete enough, timely enough, joinable enough, and evidentially healthy enough to support the control verdicts the platform will display?

This question matters directly to CES integrity. A control's score is only as reliable as the data underneath it. If `kyc_identity_verification.verified_at` is null because a vendor API timed out, the platform cannot determine whether KYC preceded activation — but it can distinguish that situation from one where KYC genuinely occurred after activation. If the platform treats both as CO-C001 failures, it will produce an inflated failure rate that misleads auditors and drives unnecessary remediation effort. If it silently drops the null-timestamp record, it produces an artificially high pass rate that exposes the bank to regulatory risk. Neither outcome is acceptable. A high CES score computed on structurally incomplete or unjoined data is not an audit asset — it is an audit liability, because it cannot survive regulatory examination.

The platform therefore carries a specific and non-negotiable obligation: it must distinguish between a control that failed to operate and a data pipeline that failed to deliver evidence of operation. These are fundamentally different conditions. The first is a compliance problem. The second is a data engineering problem. Conflating them — by allowing pipeline failures to register as control failures — destroys auditor confidence in the dashboard and makes it impossible for a control owner to understand whether they have a process problem or a technology problem. Conversely, allowing pipeline failures to register as silent passes is a regulatory risk of the first order.

To enforce this distinction, the platform uses exactly five outcome labels, each of which carries a specific audit meaning, a specific CES treatment, and a specific remediation path. A **Data Gap** means the evaluation cannot be made because required source records are absent or unjoinable — the instance is excluded from the CES denominator until data is recovered. An **Evidence Gap** means the control logic passes but no linked evidence artifact can be located to substantiate it — the instance counts as operating but degrades Evidence Completeness. A **Control Failure** means the source data affirmatively proves the control did not operate — the instance counts as a fail in the operating rate. A **Correlation Warning** means a join succeeded but at lower confidence than required for a definitive verdict — the instance is routed to human review and excluded from CES until resolved. A **Needs Review** means an ambiguous data condition requires manual analyst judgment before any classification can be assigned. These labels are not interchangeable, and this document enforces their distinctions throughout every section.

Before any control box on the Live RCM dashboard is permitted to display green, amber, or red, five conditions must be confirmed. Source data must be available and extractable from the originating system within its documented refresh window. Join keys must resolve — meaning `application_id`, `vendor_request_id`, and `loan_application_id` must be present and consistent across every system involved in the control evaluation. Timestamps must be valid, non-null, timezone-normalized, and sequenced correctly relative to the process closure event. Evidence must be linked to the correct ControlInstance with a valid hash, an upload date within the remediation window, and a `linked_control_id` that matches the control being evaluated. And the Data Readiness Score for the affected process must clear its minimum threshold before CES values are presented as reliable. This document provides the validation rules, scoring model, checklists, and issue-handling workflows needed to enforce all five conditions in the prototype and in production.

## 3. Data Gap vs Evidence Gap vs Control Failure — Definitive Distinction

### 3.1 Formal Definitions

**Data Gap** means the platform cannot evaluate a ControlInstance because one or more required source records are absent, unjoinable, or contain null values in fields that are essential to the evaluation logic. The data condition that produces it is a missing row, a null required field, a failed API response, or a join that returns no matching record. For CES computation, Data Gap instances are excluded from both the numerator and the denominator of all three sub-scores — they are not counted as passes, not counted as fails, and the CES denominator shrinks accordingly. If the Data Gap rate exceeds 30% of the expected population for any control, CES is suppressed entirely for that control.

**Evidence Gap** means the control's source logic evaluates to pass — the required action was completed, the timestamps are valid, the outcome is correct — but no linked EvidenceArtifact can be found in the document repository to substantiate the verdict. The data condition is a missing, late, orphaned, or hash-invalid document in `document_evidence_store` for the relevant `linked_control_id`. For CES computation, Evidence Gap instances are counted in the Operating Rate denominator and numerator (they pass the operating test) but their contribution to the Evidence Completeness sub-score is zero or reduced. The CES is computed, but the Evidence Completeness component is degraded.

**Control Failure** means the source data affirmatively proves that the control did not operate as designed — a required step was missing, a timestamp proves the wrong sequence, a value proves the wrong outcome. The data condition is a present, joined, valid source record whose field values satisfy the fail condition defined in the ControlInstance generation logic. For CES computation, the instance is counted in the Operating Rate denominator but not in the numerator — it reduces the operating rate. A control issue is created and routed to the responsible control owner and audit lead.

**Correlation Warning** means a join between two source records succeeded, but at a confidence level below the threshold required for a definitive evaluation verdict. The data condition is typically a secondary-key or timestamp-window join rather than a primary-anchor-key join, or a manual timestamp override that cannot be verified against a system-generated event. For CES computation, Correlation Warning instances are excluded from the CES denominator and routed to a Needs Review queue — they are neither counted as passes nor as fails until a human analyst resolves the ambiguity.

**Needs Review** means an ambiguous data condition — a duplicate record, an orphan, an unreadable document, an unresolved scope question — requires manual analyst judgment before any of the four other classifications can be assigned. The data condition may be any structural anomaly that the automated validation rules flag but cannot resolve deterministically. For CES computation, Needs Review instances are excluded from all CES sub-score calculations until the analyst classifies them and closes the review record.

| Classification | Meaning | Source Data Exists? | Evidence Exists? | Can CES Compute? | CES Treatment | Auditor Interpretation |
|---|---|---|---|---|---|---|
| Data Gap | Required source record absent, null, or unjoinable | No (or partially) | No | Only if gap rate ≤ 30% | Excluded from numerator and denominator of all sub-scores | Cannot make a verdict — data pipeline failure, not a control failure |
| Evidence Gap | Source logic passes but no linked evidence artifact found | Yes | No | Yes | Counted in Operating Rate; Evidence Completeness = 0 for instance | Control likely operated; proof is missing — records management remediation required |
| Control Failure | Source data proves control did not operate | Yes | May or may not exist | Yes | Counted in denominator; not in numerator — reduces Operating Rate | Confirmed compliance failure — control issue created and escalated |
| Correlation Warning | Join succeeded at low confidence — verdict unreliable | Partially | May exist | No | Excluded from denominator; routed to Needs Review | Data ambiguity — human analyst must confirm before audit verdict is issued |
| Needs Review | Structural anomaly requires human classification before any label is assigned | Unknown | Unknown | No | Excluded from all CES calculations until resolved | Cannot determine nature of problem — investigation required before CES can include this instance |

## 7. Data Readiness Score

### 7.1 Formula and Component Definitions

**Data Readiness Score = (Source Availability × 25%) + (Join Key Completeness × 20%) + (Timestamp Completeness × 20%) + (Evidence Linkage × 15%) + (Population Coverage × 10%) + (Data Quality Penalty Adjustment × 10%)**

| Component | Weight | What It Measures | Numerator | Denominator | Data Inputs Required | Score = 100 When | Score = 0 When |
|---|---|---|---|---|---|---|---|
| Source Availability | 25% | Whether all required source systems delivered non-empty extracts within their documented refresh SLA | Count of source systems that delivered a non-empty, on-time extract | Total required source systems for the process | Source system extract logs, row count records, extract timestamp | All required source systems delivered non-empty extracts within SLA for the measurement window | All required source system extracts are empty or missing |
| Join Key Completeness | 20% | Whether the primary anchor key (`application_id`, `vendor_request_id`, or `loan_application_id`) is non-null and consistent across all required source tables | Count of ProcessExecutions where the anchor key resolved successfully across all required source systems | Total ProcessExecutions expected for the period | Primary key field values across all source tables; cross-system key reconciliation output | Every ProcessExecution has a fully resolved anchor key across all required source systems | No ProcessExecution has a resolvable anchor key |
| Timestamp Completeness | 20% | Whether all required timestamp fields are non-null, timezone-normalized, and sequenced correctly for control evaluation | Count of ProcessExecutions where all required timestamps are non-null, UTC-normalized, and pass sequence ordering checks | Total ProcessExecutions in scope | All required timestamp fields from each source system; UTC normalization audit; sequence validation output | All required timestamps present, UTC-normalized, and correctly ordered for every ProcessExecution | All required timestamps are null or fail sequence checks |
| Evidence Linkage | 15% | Whether ControlInstances that fired have at least one linked EvidenceArtifact with a valid hash and `uploaded_at` within the allowed window | Count of ControlInstances with at least one Complete or LATE_BUT_VALID evidence record | Total ControlInstances that fired (excluding Not Applicable) | `document_evidence_store` rows with non-null `linked_control_id`, valid `hash_value`, and `uploaded_at` within remediation window | Every fired ControlInstance has at least one Complete evidence record uploaded before process closure | No ControlInstance has any linked evidence |
| Population Coverage | 10% | Whether the extracted population matches the expected population derived from the anchor system | Count of ProcessExecutions in extract | Expected ProcessExecution count from anchor system (CRM activated count, vendor approval event count, booking event count) | Anchor system count; extract row count; reconciliation report | Extracted population equals 100% of expected population | Extracted population is 0% of expected population |
| Data Quality Penalty Adjustment | 10% | Deduction applied based on the proportion of ProcessExecutions affected by validation rule failures, Correlation Warnings, or Needs Review flags | 100 minus the percentage point deduction equal to the proportion of ProcessExecutions with active validation failures × a severity weighting factor | Not applicable — this is a deduction component | Validation rule failure counts from Section 6; severity classifications; active Correlation Warning count | Zero validation failures, zero Correlation Warnings, zero Needs Review flags | All ProcessExecutions have at least one active validation failure |

### 7.2 Readiness Bands

| Band | Score Range | Interpretation | CES Treatment | UI / Product Label |
|---|---|---|---|---|
| Ready | 90–100 | All source systems available, keys resolve, timestamps valid, evidence well-linked, population complete | CES is computed and displayed as authoritative; Green / Amber / Red verdicts are trusted | **Data Ready** — CES scores are reliable |
| Usable with Warnings | 75–89 | Minor gaps in evidence linkage, population coverage, or validation failures — CES computable but caveated | CES is computed and displayed with a data quality caveat flag; scores should be reviewed alongside the Data Readiness detail panel | **Data Usable** — CES computed with caveats; review Data Readiness detail |
| Limited Trust | 60–74 | Material gaps in join key completeness, timestamps, or source availability — CES is indicative only | CES is computed but labelled as indicative; Operating Rate and Catch Rate sub-scores shown with explicit limitation; Evidence Completeness likely degraded | **Limited Trust** — CES is indicative only; significant data gaps present |
| Insufficient Data | Below 60 | CES suppressed — data quality failures too significant to support reliable control verdicts | CES is not computed; all control boxes show "Insufficient Data" regardless of any partial source records available | **Insufficient Data** — CES suppressed; data quality remediation required before scores are computed |

### 8.2 CES Suppression Decision Rules

| Rule ID | Condition | Action | Rationale |
|---|---|---|---|
| SUPP-001 | Data Readiness Score for a process is below 60 | Suppress CES for all controls in that process; display "Insufficient Data" label; do not display Green / Amber / Red | A Data Readiness Score below 60 indicates that source availability, join key completeness, or timestamp validity failures are too significant to produce a defensible control verdict; publishing CES scores from such a population creates audit liability |
| SUPP-002 | Data Gap count for any individual control exceeds 30% of that control's expected ControlInstance population for the measurement period | Suppress CES for that specific control; display "Insufficient Data" for that control only; other controls in the same process may still compute | A Data Gap rate above 30% means more than a third of the evaluable population is missing — the operating rate and catch rate computed on the remaining 70% do not reliably represent the true control performance across the full population |
| SUPP-003 | Source data confirms control logic passes (all required timestamps valid and sequenced, outcome fields correct) but no EvidenceArtifact is linked to the ControlInstance | Compute CES; do not suppress; set Evidence Completeness contribution for this instance to 0; Operating Rate and Catch Rate contributions are unaffected | The absence of evidence is an evidence management failure, not a control failure; suppressing CES would conflate the two and incorrectly penalize the Operating Rate for a records management problem |
| SUPP-004 | Source data affirmatively proves that the control did not operate (timing test fails, required field value proves non-operation, required action absent) | Compute CES; include instance in Operating Rate denominator but not numerator; reduce Operating Rate accordingly; create a control issue | A confirmed control failure must be counted; suppressing or excluding it would inflate the CES and obscure a compliance problem that requires remediation |
| SUPP-005 | A ControlInstance's join was completed using only a secondary key or timestamp window, and the join confidence is flagged as LOW (Correlation Warning) | Exclude instance from CES denominator and numerator for all sub-scores; route instance to Needs Review queue | A low-confidence join cannot support a reliable verdict in either direction; including it in the CES would credit or penalize the control based on data that may belong to a different ProcessExecution |
| SUPP-006 | A duplicate process execution is detected (two ProcessExecution records with the same real-world business event) and has not been resolved by analyst deduplication | Exclude all ControlInstances from both duplicate ProcessExecutions from CES denominator and numerator until deduplication is complete | Including both duplicates double-counts the event and splits evidence across two executions; the combined CES impact is unreliable until it is known which ProcessExecution is canonical |
| SUPP-007 | An EvidenceArtifact linked to a ControlInstance has a `hash_value` that does not match the current file hash (invalid hash) | Set Evidence Completeness contribution for this instance to 0; do not exclude the instance from CES denominator; create an evidence integrity investigation record | An invalid hash indicates possible file replacement or corruption; the evidence cannot be relied upon to substantiate the control verdict; however, the source-logic verdict (pass or fail) remains valid and the instance should remain in the denominator |
| SUPP-008 | The total number of ControlInstances that fired for a given control in the measurement period is fewer than 10 | Do not compute CES percentage values for that control; display raw instance counts and outcome labels instead; label as "Insufficient Population" | Computing a percentage from fewer than 10 instances produces statistically unreliable scores; a single instance flip changes the CES by more than 10 percentage points; percentage presentation misleads rather than informs |
