# Report Page: SWIFT Auditor Document (AI-Generated)

Plan for turning the Report page into an **auditor-style document**: the user has all assessment results in hand, and the system produces a **single, coherent report document** generated by the **same AI model** used for evidence evaluation (Vertex AI / Gemini).

---

## 1. Vision

- **Role:** The Report page is the place where a **SWIFT auditor** (or equivalent) sees the full picture and produces the **assessment report as a document**.
- **Content:** The report is a **single document** with standard sections (Executive Summary, Scope, Methodology, Control Assessments, Gap Analysis, Evidence Index, Glossary). Each section is **AI-generated prose** based on real cycle data, using the same tone and rigour as the evidence evaluator (“You are a SWIFT CSCF v2025 compliance auditor”).
- **Model:** Use the **same Vertex AI model** as evidence evaluation (`settings.VERTEX_AI_MODEL` / `ai_service._get_model()`), so behaviour and quality are consistent across evaluation and reporting.

## 3. Data: “All Results in the Auditor’s Hand”

Before generating the document, the backend builds a **report snapshot** that represents “all results” for the cycle. The AI will receive this (or a subset per section) so the document is grounded in real data.

**Snapshot contents (conceptual; store in `assessment_reports.snapshot_data`):**

- **Cycle:** `cycle_id`, `label`, `cycle_year`, `architecture_type`, `phase`.
- **Domains:** List of domains A–H with `id`, `name`, `sort_order` (from `evidence_domains`).
- **Domain scores:** Per domain: `id`, `name`, `completed`, `total`, `score` (same as dashboard).
- **Control scores:** Per applicable control: `control_id`, `name`, `type` (M/A), `score`, `status`, `evidence_count`; optionally grouped by domain for “Control Assessments”.
- **Gaps:** List of controls below threshold (e.g. score &lt; 50, mandatory): `control_id`, `name`, `score`, optional `domain_id`.
- **Evidence index:** Per evidence item (or per domain): `evidence_item_id`, `name`, `domain_id`, submission `status` (draft/submitted/in_review/approved).
- **Approval state:** Gate statuses (evidence_complete, review_complete, gaps_documented, final_attestation) and overall compliance %.
- **Sufficiency summaries (optional):** Per control or per evidence item, high-level “met / not met” or score from `sufficiency_scores` / evaluations, so the AI can refer to “evaluation results” in the narrative.

Implementation: reuse dashboard API logic and approval summary logic; add a small **report snapshot builder** (e.g. in `backend/app/services/report_snapshot.py` or inside reports router) that assembles the above and returns a dict (and optionally writes it to `report.snapshot_data`).

## 4. AI Report Generation (Same Model as Evaluation)

### 4.1 Reuse and extend AI service

- **Reuse:** Same Vertex AI client and model as evidence evaluation:
  - Use `_get_model()` (or a thin wrapper) so the report uses **exactly** `settings.VERTEX_AI_MODEL` (e.g. `gemini-2.5-flash-lite`).
- **New:** A dedicated **report generation** entrypoint that:
  - Takes **section name**, **snapshot (or section-specific context)**, and optionally **previously generated section texts** (so later sections can reference “as stated in the Executive Summary”, etc.).
  - Sends a **report-specific prompt** (see below) and gets back **plain text** (prose), not JSON.
  - Handles rate limits and errors the same way as `evaluate_evidence` (retries, clear error messages).

Suggested signature:

```text
generate_report_section(
    section_name: str,
    section_instruction: str,
    snapshot_context: str,
    previous_sections: list[tuple[str, str]] | None = None,
) -> str
```

- **Prompt file:** Add a new prompt template under the same `Prompt` directory, e.g. `report_generation_V1.txt`, so report behaviour is editable without code changes.

### 4.2 Report prompt design (auditor persona, same model)

- **Persona:** Start with the same as evidence evaluation: “You are a SWIFT CSCF v2025 compliance auditor.”
- **Task:** “Write the [section name] of an assessment report. Use ONLY the following assessment data. Write in clear, formal prose suitable for an official assessment report. Do not invent data.”
- **Input:** Include:
  - **Structured snapshot:** A text representation of the snapshot (e.g. JSON or key–value blocks) so the model sees cycle, domains, control scores, gaps, evidence index, approval state.
  - **Section-specific instruction:** e.g. “Executive Summary: summarise overall compliance, architecture, number of gaps, and key findings in 1–2 short paragraphs.” “Control Assessments: for each domain, list controls and their scores; note any below threshold.” “Gap Analysis: list identified gaps and note remediation or risk acceptance if present.”
- **Optional:** Short “previous sections” summary (e.g. “Executive Summary stated: …”) so the full document reads as one coherent report.
- **Output:** Plain text only (no JSON, no markdown fences unless you want headings). The backend stores this in the section’s `content` field.

Example structure for the prompt file:

```text
You are a SWIFT CSCF v2025 compliance auditor.

Write the following section of an assessment report. Use ONLY the assessment data provided below. Write in clear, formal prose suitable for an official assessment report. Do not invent or assume any data.

## Section to write: {section_name}

## Your instructions for this section:
{section_instruction}

## Assessment data (use only this):
{snapshot_context}

{previous_sections_block}

Output ONLY the section text (prose). No JSON, no meta-commentary. Use headings only if the section requires subsections (e.g. by domain).
```

### 4.3 Section-by-section generation

- Generate **one section at a time** (to stay within token limits and to allow per-section retry).
- Order: e.g. Executive Summary → Scope Statement → Methodology → Control Assessments → Gap Analysis → Evidence Index → Glossary.
- For each section:
  - Build `section_instruction` and, if needed, a reduced `snapshot_context` (e.g. for “Evidence Index” only domains + evidence list; for “Gap Analysis” only gaps + remediation info).
  - Optionally pass `previous_sections` (list of (section_name, content)) so the model can keep the narrative consistent.
  - Call `generate_report_section(...)` and store the returned string in `sections[i].content`.
- After all sections, persist `sections` and `snapshot_data` to the report (same `assessment_reports` row).

## 5. API Design

- **Build snapshot (no AI):**  
  `POST /api/v1/assessments/{cycle_id}/reports/{report_id}/build_snapshot`  
  Builds snapshot from DB, writes to `report.snapshot_data`, returns snapshot (or 204). Optional: create report if missing.

- **Generate full report (AI):**  
  `POST /api/v1/assessments/{cycle_id}/reports/{report_id}/generate`  
  1) Ensure snapshot exists (build if not).  
  2) For each section that is “AI” (e.g. Executive Summary, Control Assessments, Gap Analysis), call `generate_report_section` and set `sections[i].content`.  
  3) For non-AI sections (Scope, Methodology, Evidence Index, Glossary), either leave empty for manual edit or fill from a simple template (e.g. Scope = “Architecture: {arch}, Domains: A–H”).  
  4) Save `sections` and return updated report.  
  Consider async or polling if generation takes long (e.g. 30+ seconds).

- **Regenerate one section:**  
  `POST /api/v1/assessments/{cycle_id}/reports/{report_id}/sections/{section_index}/regenerate`  
  Re-run AI for that section only; update `sections[section_index].content` and save.

- **GET/PUT report:** Keep existing `GET` and `PUT` so the frontend can load the document and save edits. Ensure `sections[].content` is included in response and in PUT body.

- **Finalize / Export:** Keep existing finalize; add export (PDF/Word) that renders `sections[].content` as the report body.

## 6. Report Page UX (Auditor “All Results in Hand”)

- **Single-document view:** Present the report as **one scrollable document** (all sections in order), not only one section at a time. The auditor should see the full report like a Word/PDF.
- **“All results in hand”:** Before generating, show a short **summary of inputs** (e.g. “Cycle X, architecture A2, 32 controls, 5 gaps, 48/53 evidence items approved”) so the user confirms the data the AI will use.
- **Generate / Refresh:** One primary action: **“Generate report”** or **“Refresh from assessment”** that calls `POST .../generate`. Show loading (e.g. “Generating report…”, optionally “Section 1/7…”). When done, show the full document with all section contents.
- **Per-section:** Each section shows its **AI-generated content** (read-only by default). Actions: **Edit** (inline or modal), **Regenerate** (call regenerate endpoint for that section). Edits saved via existing PUT.
- **Finalize and export:** **Finalize report** (existing API) then **Export PDF / Word** so the auditor can download the final document.

## 7. Implementation Order

1. **Snapshot builder** – Build snapshot dict from cycle (dashboard + approval + domains + gaps + evidence index). No AI. Optionally expose via `build_snapshot` or do it inside `generate`.
2. **Report prompt** – Add `report_generation_V1.txt` with persona, placeholders for `section_name`, `section_instruction`, `snapshot_context`, `previous_sections_block`.
3. **AI service** – Add `generate_report_section(...)` in [ai_service.py](backend/app/services/ai_service.py) using same `_get_model()`, load report prompt, call `generate_content`, return text (no JSON parse).
4. **Generate endpoint** – `POST .../reports/{report_id}/generate`: build snapshot, loop over sections, generate AI sections, fill template for non-AI sections, save.
5. **Regenerate section endpoint** – `POST .../reports/{report_id}/sections/{section_index}/regenerate`.
6. **Frontend** – Report page: show “inputs” summary, “Generate report” button, full-document view with section content, Edit/Regenerate per section, Finalize, Export.

## 9. Summary

- **Report = SWIFT Auditor document:** One coherent report document, all results in hand (snapshot), AI-generated narrative.
- **Same model as evaluation:** Use the same Vertex AI model (Gemini) via `ai_service._get_model()` and a new report prompt file.
- **Flow:** Build snapshot → generate each section with `generate_report_section(section_name, instruction, snapshot, previous_sections)` → store in `sections[].content` → show as single document; user can edit, regenerate section, finalize, export.

This keeps the Report page aligned with the evidence evaluation quality and persona while delivering a single, auditor-style assessment report.
